### Mandatory lifecycle (after initialization gate completes):

1. **Ambient Preflight** — read context capsule, task memory, protocol files, git status.
2. **Solution Formation** — understand request, diagnose, form concrete solution.
3. **User Confirmation** — present solution, wait for explicit "方案 OK".
4. **Task-Card Instruction Gate** — user must explicitly issue task-card instruction
   before routing. "方案 OK" alone ≠ task card approval.
5. **Execution Contract → Value Route → Routing** — `ags_solution_check` surfaces a Value Route (效价比路由): the minimal execution-path form that still covers the risk (advisory; does NOT change level/permission/gates). Then classify as Light/Medium/Heavy from the confirmed solution.
6. **Gate / Execution / Receipt** — validate → resolve → execute → verify → deliver.

Existing-card entry: if the first non-empty line is `## 任务卡`, do not run the
raw-request lifecycle again. `ags_solution_check` validates first; a valid card
continues to `ags_policy_resolve` and the runner, while an invalid card stops.

Task cards have exactly two permission modes: `plan-only` and
`execute-and-verify`. Light/Medium default to direct execution. Heavy defaults
to `plan-only`, but explicit Heavy `execute-and-verify` runs and verifies
directly; Heavy adds an independent review gate, not a plan-first or extra
permission pause. Protected destructive/external/release actions keep their
own stop conditions.
