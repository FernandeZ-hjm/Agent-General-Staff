## Task-Card Handoff Gate

The solution has been confirmed, but `方案 OK` does not create a task-card
handoff. The host must provide structured handoff evidence to the Request Router.

### Direct execution

An explicit same-session modification instruction such as `开改`, `按这个改`, or
`开始实现` authorizes host-native `direct-edit`. Do not compile a task card.
Independent protected-path, release, external-write, credential, migration,
destructive-operation, review, and verification boundaries still apply.

### Task-card handoff

Generate a task card only after an explicit handoff instruction such as:

- `生成任务卡`
- `按这个方案出任务卡`
- `交给 Claude Code 执行`

The solution/diagnosis, scope, verification, and handoff boundaries must also be
confirmed through structured host state. The router/compiler blocks card output when:

- the instruction is missing: `block_reason: task_card_not_requested`
- the confirmed contract is missing: `block_reason: handoff_contract_not_confirmed`

The compiler gate protects handoff artifact generation. It is not a prerequisite
for authorized direct local execution.

### Existing task cards

If the first non-empty line is `## 任务卡`, validate it as an existing handoff
contract. Valid cards skip generation and continue to policy/runner; invalid
cards fail closed.

### Handoff sequence

1. Confirm solution/diagnosis, scope, verification, and handoff boundaries.
2. Receive an explicit task-card/handoff instruction.
3. Pass both structured gates.
4. Classify Light/Medium/Heavy from the confirmed contract.
5. Generate the canonical task card.
6. Validate, resolve policy, and dispatch to the named executor.
